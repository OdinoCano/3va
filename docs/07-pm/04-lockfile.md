# 04 - LOCKFILE FORMAT

## 4.1 Overview

The `3va-lock.json` lockfile ensures reproducible installations, per-package origin traceability, and security audits. It is automatically generated by `3va install` and `3va reinstall`, and is read by `3va update`.

**Key principle:** the lockfile is the source of truth for knowing which version of each package is installed *and from which registry it comes*. This allows `3va update` to respect origins without silent network calls.

---

## 4.2 JSON Format

```json
{
  "lockfileVersion": 3,
  "name": "my-project",
  "version": "1.0.0",
  "packages": {
    "": {
      "version": "1.0.0",
      "resolved": null,
      "integrity": null,
      "dev": null
    },
    "node_modules/axios": {
      "version": "1.7.2",
      "resolved": "https://registry.npmjs.org/axios/-/axios-1.7.2.tgz",
      "integrity": "sha512-...",
      "registry": "registry.npmjs.org"
    },
    "node_modules/@std/path": {
      "version": "0.196.0",
      "resolved": null,
      "integrity": null,
      "registry": "jsr.io"
    }
  },
  "dependencies": {
    "axios": {
      "version": "1.7.2",
      "resolved": "https://registry.npmjs.org/axios/-/axios-1.7.2.tgz",
      "integrity": "sha512-...",
      "registry": "registry.npmjs.org"
    },
    "@std/path": {
      "version": "0.196.0",
      "resolved": null,
      "integrity": null,
      "registry": "jsr.io"
    },
    "follow-redirects": {
      "version": "^1.15.0",
      "resolved": null,
      "integrity": null
    }
  }
}
```

---

## 4.3 Root Fields

| Field | Type | Description |
|-------|------|-------------|
| `lockfileVersion` | `number` | Format version. Currently `3`. |
| `name` | `string` | Project name (from `package.json`). |
| `version` | `string` | Project version. |
| `packages` | `object` | Map of package entries including `node_modules/*`. |
| `dependencies` | `object` | Resolved top-level dependencies. |

---

## 4.4 Package Entry (`LockfilePackage`)

Appears under `packages["node_modules/<name>"]`:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | `string` | Yes | Exact installed version. |
| `resolved` | `string \| null` | No | Full URL of the downloaded tarball. |
| `integrity` | `string \| null` | No | Integrity hash (sha512 base64 or sha256). |
| `dev` | `boolean \| null` | No | `true` if it is a development dependency. |
| `registry` | `string \| null` | No | Source registry host (e.g., `registry.npmjs.org`, `jsr.io`). Omitted if unknown. |

---

## 4.5 Dependency Entry (`LockfileDep`)

Appears under `dependencies["<name>"]`:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | `string` | Yes | Installed version. |
| `resolved` | `string \| null` | No | Tarball URL. |
| `integrity` | `string \| null` | No | Integrity hash. |
| `dependencies` | `object \| null` | No | Transitive subdependencies. |
| `dev` | `boolean \| null` | No | Development dependency. |
| `registry` | `string \| null` | No | Source registry host. This field is what `3va update` uses to know where to connect. |

---

## 4.6 `registry` Field â€” Origin Tracking

### 4.6.1 Purpose

In large projects, different packages may come from different registries:

```json
"axios":    { "version": "1.7.2",   "registry": "registry.npmjs.org" },
"react":    { "version": "18.3.1",  "registry": "registry.yarnpkg.com" },
"@std/path":{ "version": "0.196.0", "registry": "jsr.io" }
```

The `registry` field is the source of truth for `3va update`: it guarantees that each package is updated from its original registry without silent origin changes.

### 4.6.2 When it is written

- It is written on each `3va install` or `3va reinstall` for the newly installed package.
- It is preserved for other packages when regenerating the lockfile (the generation process loads the previous lockfile and copies existing values before overwriting).
- **It is not written** for transitive dependencies inferred by the resolver (no real registry was contacted for them).

### 4.6.3 Recognized Values

| Value | Registry |
|-------|----------|
| `registry.npmjs.org` | Official npm |
| `registry.yarnpkg.com` | Yarn |
| `jsr.io` | JSR (JavaScript Registry) |
| Any other host | Custom registry derived from the `--allow-net` host |

### 4.6.4 Registry Migration

To migrate a package to another registry, use `install` with the new `--allow-net`. This overwrites the `registry` field in the lockfile explicitly and auditable:

```bash
# axios will now be updated from Yarn in the future
3va install axios --allow-net=registry.yarnpkg.com
```

## 4.7 Integrity and Security

### 4.7.1 Integrity Hash

```
integrity: sha512-<hash-base64>
```

Calculated with `SignatureVerifier` (SHA-256 or SHA-512 depending on configuration).

### 4.7.2 Verification

```bash
# Verification happens automatically on install/reinstall/update
3va install axios --allow-net=registry.npmjs.org
# â†’ [SIGNATURE] Verifying axios@1.7.2 ...
```

## 4.8 Operations

### 4.8.1 Generate / Update Lockfile

```bash
# Install and record origin
3va install axios --allow-net=registry.npmjs.org
# â†’ writes registry: "registry.npmjs.org" to 3va-lock.json

3va install @std/path --allow-net=jsr.io
# â†’ writes registry: "jsr.io"; preserves axios's registry
```

### 4.8.2 Update from Lockfile

```bash
# Check which registries are needed before running
3va update
# âś— Update requires network access to:
#     registry.npmjs.org   (axios)
#     jsr.io               (@std/path)
# Run: 3va update --allow-net=registry.npmjs.org,jsr.io

# Run the update
3va update --allow-net=registry.npmjs.org,jsr.io

# Update only one package
3va update axios --allow-net=registry.npmjs.org
```

### 4.8.3 Reinstall

```bash
3va reinstall axios --allow-net=registry.npmjs.org
```

---

## 4.9 Compatibility

The format is compatible with npm's `package-lock.json` v3 in the base fields (`version`, `resolved`, `integrity`, `dependencies`). The `registry` field is a 3va-specific extension; npm tools will ignore it.
